# Experienced-Level Interview Panel Q&A Guide: HRMS Portal

This document serves as an end-to-end technical reference answering core architecture, implementation, database schemas, geofencing coordinates, Redux stores, and security questions about this Employee Management System.

---

## 1. System Design & Architecture

### Core Questions

* **Can you walk us through your HRMS system architecture end-to-end?**
  The system is built on a decoupled, three-tier architecture:
  1. **Presentation Layer (Frontend)**: Built with React, Redux Toolkit (state management), Tailwind CSS (styling), and React Router DOM. It operates as a Single Page Application (SPA).
  2. **Application Layer (Backend)**: Built with Node.js and Express. It functions as a stateless RESTful JSON API using middleware stacks for authentication (JWT decoding), role auditing, and Zod query/body validations.
  3. **Database Layer**: Built with MongoDB and Mongoose ODM. It runs on a document model schema with custom validation hooks (like automated password hashing and unique identity code generation before document commits).

* **How does data flow from frontend → backend → database?**
  1. **Event Trigger**: A user interacts with the UI (e.g. clicks "Clock In").
  2. **Validation**: The frontend validates input constraints (e.g. latitude is numeric) using Zod.
  3. **API Request**: Axios makes an asynchronous request to the backend carrying a JWT access token in the `Authorization` header.
  4. **Routing & Middlewares**: Express intercepts the request. The request passes through:
     - `morgan` (logging HTTP details).
     - `cors` (validating origin).
     - `protect` middleware (verifies JWT token signature and populates `req.user` and `req.company` payloads).
     - `validate` middleware (Zod schema checking).
  5. **Controller Processing**: The controller executes business logic (e.g. calculating Haversine geofenced bounds).
  6. **Database Operation**: Mongoose writes/reads from MongoDB.
  7. **HTTP Response**: The controller returns a JSON payload containing the database document and a success indicator back to the client.
  8. **State Dispatch**: Redux catches the response data and updates state. The React UI re-renders to reflect the change.

* **How did you design your system to support multiple roles in one application?**
  The application utilizes a single codebase with dynamic rendering based on active session attributes:
  - **Database Level**: The `User` model stores a `role` field restricted by Mongoose enums (`["employee", "owner"]`).
  - **Backend Access Control**: Restricts entire Express route scopes using an `isAuthorized(...allowedRoles)` helper.
  - **Frontend State**: The Redux auth slice (`state.auth.role`) caches the active user's role.
  - **UI Router**: Route paths check the user's role state, either mapping them to `/employee` paths, `/owner` paths, or redirecting to `/unauthorized` if they attempt unauthorized path configurations.

### Deep Push

* **How do you ensure separation of concerns between roles?**
  1. **Strict Backend Middleware Barriers**: APIs are locked by role validations. For example, `router.post("/add", isAuthorized("owner"), addUser)` makes it impossible for standard employees to request employee creations, even if they spoof parameters.
  2. **Data Scope Masking**: Controllers verify the requesting user's identity. In the leave controller, standard employees only search leaves containing `{ user: req.user._id }`, while owners fetch company-wide leaves `{ company: req.company._id }`.
  3. **Conditional Frontend Navigation**: Dashboards and sidebar links are conditionally generated by checking `role === 'owner'`.

* **If 500 companies use your HRMS (multi-tenant system), how will your design change?**
  The codebase is already designed around a shared-database, multi-tenant layout. Every document (`User`, `Attendance`, `Leave`, `Payroll`) has a `company` field referencing the `Company` collection.
  For 500 companies, we would scale by:
  1. **Database Indexes**: Creating compound index constraints like `{ company: 1, user: 1 }` and `{ company: 1, email: 1 }` to ensure queries remain efficient.
  2. **Tenant Routing Subdomains**: Routing tenants via unique subdomains (e.g. `companyA.hrms.com`, `companyB.hrms.com`). The frontend resolves the tenant from the URL, calls public info `/company/public/:tenantId`, and dynamically configures branding.
  3. **Database Sharding**: As write rates increase, we would shard the MongoDB cluster using `companyId` as the shard key, ensuring all data related to a single company resides on the same shard.

### Killer Question

* **Is your system single-tenant or multi-tenant? How would you convert it?**
  Our system is a **multi-tenant system using a single-database shared-schema model**. The conversion of a single-tenant system (isolated database per tenant) to this model or vice-versa involves:
  - **Converting to Shared-Schema Multi-Tenant**: Injects a mandatory `company` tenant ID pointer on every single model. The backend must enforce that every database operation is scoped by the tenant identifier (e.g. `User.findOne({ email, company: req.company._id })`).
  - **Converting to Database-per-Tenant Model**:
    1. The backend establishes connections dynamically based on the tenant resolver (e.g. checking headers or hostnames).
    2. Connection pools are cached inside memory: `const tenantDB = connections[tenantId] || mongoose.createConnection(...)`.
    3. The model schemas are compiled dynamically against the connection pool instead of the global database instance.

## 2. Authentication & Role-Based Access Control (RBAC)

### Core Questions

* **Explain your authentication flow (login, register, OTP).**
  1. **Registration Request (`send-otp`)**: Owner enters corporate email and company name. The server inserts unverified `Company` and `User` documents, generates a random 5-digit OTP code, sets `otpExpiry` (now + 15m), and sends an email.
  2. **OTP verification (`verify-otp`)**: Owner enters the received code. The backend verifies the token and sets `isVerified = true` in both collections.
  3. **Onboarding completion (`register`)**: Owner inputs their full name and desired password. The database hashes the password, completes onboarding, and emails success.
  4. **Login (`login`)**: User submits credentials. The backend verifies the email and uses `bcrypt.compare` to check the password. On success, it issues a JWT access token and sets a secure `httpOnly` refresh token cookie.

* **How do you assign roles (Owner, Admin, Manager, Staff)?**
  - Our system is structured around two key roles: `owner` (full organizational privileges) and `employee`.
  - When the owner adds a user via `POST /api/users/add`, the backend defaults the role to `"employee"`.
  - The owner can update roles of users in the company to different profiles or administrative statuses using the admin-update endpoint.

### RBAC Focus

* **How do you restrict routes based on roles?**
  On the backend, we use the `isAuthorized` middleware:
  ```javascript
  const isAuthorized = (...roles) => (req, res, next) => {
      const allowedRoles = roles.length > 0 ? roles : ['owner'];
      if (allowedRoles.includes(req.user.role)) {
          next();
      } else {
          return res.status(403).json({ success: false, message: 'Insufficient permissions' });
      }
  };
  ```
  On the frontend, routes are wrapped inside `ProtectedRoute.jsx`:
  ```javascript
  const ProtectedRoute = ({ children, allowedRoles }) => {
      const { token, role } = useSelector((state) => state.auth);
      if (!token) return <Navigate to="/login" replace />;
      if (allowedRoles && !allowedRoles.includes(role)) return <Navigate to="/unauthorized" replace />;
      return children;
  };
  ```

* **How do you protect backend APIs from unauthorized access?**
  1. **Signature Checks**: The `protect` middleware extracts the JWT from the `Authorization: Bearer <token>` header and verifies it using `jwt.verify(token, process.env.JWT_TOKEN)`.
  2. **Scope Constraints**: Every route endpoint checks `isAuthorized(...)`.
  3. **Context Filtering**: Controllers construct DB queries scoped to the logged-in user's company context: `company: req.company._id`.

### Deep Push

* **Where is role validation happening — frontend, backend, or both?**
  **Both**.
  - **Frontend Validation**: Provides immediate UX feedback, showing/hiding buttons and blocking route changes so users don't see blank screens or error codes.
  - **Backend Validation**: Enforces security constraints. Since clients can modify local code or construct arbitrary requests, the backend does not trust any user-claimed role from the client. It only trusts the role decoded from the secure JWT token.

### Killer Question

* **What happens if a user manually changes role in frontend?**
  If a user modifies the Redux store state or local storage value to bypass frontend barriers:
  1. They might successfully view hidden admin UI panels.
  2. However, the moment the UI makes an API request to retrieve admin data or trigger actions, the request will fail.
  3. This is because the backend validates the role using the **JWT signature** decoded from the token. Since the client cannot forge the JWT signature without the server's private key (`JWT_TOKEN`), the backend rejects the request with a `403 Forbidden` error.

## 3. Organization & Multi-Level Hierarchy Design

### Core Questions

* **How did you design relationships between: Owner → Admin → Manager → Staff?**
  - **Multi-Tenant Scope**: All users are linked to a single company through `company: companyId`.
  - **Owner**: The primary administrative account created during onboarding. The `Company` document points to the owner's user record: `owner: userId`.
  - **Staff/Employees**: User records have `role: "employee"`.

### Design Thinking

* **Is hierarchy stored as parentId, separate collections, or role-based mapping?**
  It is stored as a **role-based mapping within a single collection** (`User`) where administrative boundaries are checked by `role` values and company-owner references.
  - **Benefits**: Simplifies queries, eliminates deep recursive lookups, and keeps indexes flat and fast.
  - **Alternative structures**: For deep hierarchal structures, storing a `managerId` (`parentId`) pointing to another `User` document in the same collection is standard.

### Deep Push

* **How do you fetch “All staff under a manager” efficiently?**
  If using a parent-pointer hierarchy (where users have a `managerId` field):
  1. Add a MongoDB index on `managerId`: `userSchema.index({ managerId: 1 })`.
  2. Execute query: `User.find({ managerId: managerUserId })`.
  3. For multi-level query resolution (e.g. retrieving direct and indirect reports), utilize MongoDB `$graphLookup` aggregation pipelines.

### Killer Question

* **How would you support multiple managers for a single staff?**
  To support multiple managers (e.g. in matrix organizations):
  - Change the `managerId` reference in the schema from a single ObjectId to an array of ObjectIds:
    ```javascript
    managers: [{ type: mongoose.Schema.Types.ObjectId, ref: 'User' }]
    ```
  - Index the array to maintain query performance: `userSchema.index({ managers: 1 })`.
  - Query remains simple because MongoDB automatically queries values inside arrays: `User.find({ managers: targetManagerId })`.

## 4. Database Design (MongoDB)

### Core Collections Expected

* **`users`**: Contains credential hashes, roles, base salaries, positions, branches, corporate identity codes, and OTP fields.
* **`companies`**: Contains corporate settings, office geolocation coordinates, geofence radius limits, and logo references.
* **`attendances`**: Tracks check-in and check-out times, total hours worked, and boundary notes.
* **`leaves`**: Contains leave dates, types (sick, unpaid, etc.), status flags, and reviewer remarks.
* **`payrolls`**: Holds calculated monthly salary breakdowns, taxes, leave deductions, and payment status indicators.

### Core Questions

* **Can you explain your schema design?**
  - Schemas utilize Mongoose schemas with validation constraints.
  - Relationships are mapped using references (`mongoose.Schema.Types.ObjectId`) to support relational queries via Mongoose `.populate()`.
  - Check out `backend/src/models/` for details.

* **How are relationships handled?**
  Relationships use standard **references (normalized data design)**. For example:
  - `User` references `Company` via `company: ObjectId`.
  - `Attendance`, `Leave`, and `Payroll` reference both `User` (`user: ObjectId`) and `Company` (`company: ObjectId`).
  - This allows us to perform easy Joins while avoiding document size limit boundaries.

### Critical Thinking

* **How do you maintain data consistency across roles?**
  1. **Strict Route Schemas**: Zod validation schemas strip out unconfigured properties during request parsing, preventing users from modifying attributes outside their scope.
  2. **Atomic Operations**: Operations use atomic updates where possible (e.g. `findOneAndUpdate` or transactional writes) to avoid dirty read/write cycles.

### Deep Push

* **What happens when a company has 10,000 employees?**
  1. **Indexes**: Ensure search fields like `fullName`, `email`, and `identity` are indexed.
  2. **Pagination**: Enforce query limit boundaries on all search and listing endpoints to load data in small chunks (e.g. 10 or 20 records at a time).
  3. **Projection**: Strip out unneeded payload fields like password hashes or refresh tokens: `.select("-password -refreshToken")`.

### Killer Question

* **Would you use embedding or referencing for attendance records? Why?**
  **Referencing**.
  - **Why not embedding?** Storing daily attendance records inside a nested array in the `User` document would quickly blow past MongoDB's **16MB document size limit** as employees clock in thousands of times over their careers. It would also make querying attendance data across the company slow and complex.
  - **Why referencing?** Storing attendance records in a separate collection scales efficiently and enables pagination, filtering by date ranges, and sorting.

## 5. Attendance System (Clock In / Clock Out)

### Core Questions

* **How does your clock-in / clock-out system work?**
  - **Clock In**: Receives latitude and longitude coordinates, validates them against the company's geofencing boundary, and creates a new `Attendance` record with `checkInTime = new Date()`.
  - **Clock Out**: Finds today's active attendance document and updates it with `checkOutTime = new Date()` and calculates the difference in hours.

* **How do you track working hours?**
  Working hours are calculated during clock-out:
  $$\text{Hours} = \text{parseFloat}\left(\frac{\text{checkOutTime} - \text{checkInTime}}{1000 \times 60 \times 60}\right).\text{toFixed}(2)$$
  The result is saved directly to the database in `totalHours` for payroll processing.

### Deep Push

* **What happens if a user forgets to clock out?**
  - The system will detect that a previous record is active (i.e. has a check-in time but no check-out time).
  - The background proximity checking endpoint `/verify-proximity` will automatically clock out the user if it detects they are no longer within range.
  - On the next day, since `date` comparison boundaries reset, the database query checks for records within the calendar boundaries of today, allowing a new clock-in record to be created.

* **What happens if a user clocks in twice?**
  The clock-in route prevents duplicates by querying for existing entries:
  ```javascript
  const today = new Date();
  today.setHours(0,0,0,0);
  const existingAttendance = await Attendance.findOne({
      user: userId,
      date: { $gte: today }
  });
  if (existingAttendance) return res.status(400).json({ message: "Already clocked in for today" });
  ```

### Edge Case

* **How do you handle timezone differences?**
  All timestamps are saved in **UTC format** (ISO 8601) in MongoDB. The frontend resolves UTC values into local browser timezones for display. Daily bounds are computed using UTC offsets or midnight resets based on the client's local timezone.

### Killer Question

* **How do you prevent fake attendance entries?**
  1. **Backend Proximity Checks**: The backend recalculates coordinates using the Haversine formula against company settings stored in the database.
  2. **Location Access Enforced**: If GPS permissions are denied or mock coordinates are invalid, the request is rejected.
  3. **Background Audits**: Periodic geofence audits automatically clock out users who spoof or leave coordinates range.

## 6. Leave Management System

### Core Questions

* **How does leave application flow work?**
  1. Employee submits type, start/end dates, and reason via `POST /api/leaves/apply`.
  2. A new document is saved with `status: "pending"`.
  3. The owner views pending requests via `/leaves/history` and approves or rejects the request with comments using `/api/leaves/:leaveId/status`.

* **How are approvals handled?**
  Approved status updates are processed atomically:
  ```javascript
  const leave = await Leave.findOneAndUpdate(
      { _id: leaveId, company: companyId },
      { status, remarks },
      { new: true }
  );
  ```

### Deep Push / Killer Question

* **What happens if Manager rejects but Admin approves? / Consistency in approval chain**
  - To support multi-level approvals, the `Leave` schema can incorporate an approval workflow state tracking step (e.g. `currentApprovalStep`, `approvalsStatus` map).
  - To maintain consistency:
    - Decisions are locked by sequence checks (e.g., a manager cannot review after an admin has already approved).
    - Status fields transition sequentially: `pending_manager` $\rightarrow$ `pending_admin` $\rightarrow$ `approved`.

## 7. Payroll System

### Core Questions

* **How is payroll generated?**
  - Generated sequentially month-by-month to prevent data gaps.
  - Calculations look up gross salaries and subtract unpaid leave deductions and taxes.
  - Detailed breakdown is stored in a `Payroll` document.

* **What data is required for payroll calculation?**
  - **Employee Gross Salary**: Cached on the `User` model.
  - **Days in Month**: Calculated using `new Date(year, month, 0).getDate()`.
  - **Leaves**: Approved unpaid leave records overlapping the pay period boundaries.

### Deep Push / Critical Thinking

* **What happens if payroll is edited after approval? / Prevention of changes**
  - The schema has a `status` field (`"generated"` or `"paid"`).
  - **Write Locks**: Once marked as `"paid"`, the backend rejects any updates to the payroll record.
  - **Calculation Locks**: Sequential period calculations verify past states, locking historical records from modifications.

### Killer Question

* **How would you implement audit logs for payroll changes?**
  1. Create an `AuditLog` collection: `{ action, model, targetId, changedBy, previousValues, currentValues, timestamp }`.
  2. Utilize a Mongoose pre-save hook on the `Payroll` schema to compare modified paths:
     ```javascript
     payrollSchema.pre('save', async function (next) {
         if (!this.isNew) {
             const previous = await this.constructor.findById(this._id);
             // Compare fields and insert audit log record
         }
         next();
     });
     ```

## 8. Approval Workflow System

### Core Questions / Deep Push / Killer Question

* **How do you design approval flows? Status vs Event based?**
  We utilize a **status-based transition system** (`"pending"`, `"approved"`, `"rejected"`).
  - **Multi-Level Partial Approvals**:
    - For multi-tiered workflows, store an array of reviewers and their statuses:
      ```javascript
      approvals: [{ level: Number, role: String, status: String, reviewedBy: ObjectId }]
      ```
    - The document is only marked as `"approved"` when the final level transitions successfully.

## 9. Frontend Architecture

### Core Questions

* **How did you structure your React app?**
  The app follows a feature-based folder structure:
  - `src/app/`: Core store config, routing wrappers, and Axios interceptors.
  - `src/features/`: Component directories grouped by business module (e.g., auth, attendance, leaves, payroll, profile).
  - `src/components/`: Reusable layouts and components.
  - `src/utils/`: Shared utilities (validation, notifications, calculations).

* **How do you render different dashboards for different roles?**
  We use conditional rendering and protected route wrappers checking Redux states (`state.auth.role`). Dashboard views render role-specific components (e.g. `OwnerAttendance.jsx` vs `EmployeeAttendance.jsx`) dynamically.

### State Management / Deep Push / Killer Question

* **Global vs Local State? Hiding UI security?**
  - **Global State**: Session metadata (token, user context, role) and shared cache parameters are stored in Redux.
  - **Local State**: Input fields, validation errors, and toggle flags are stored in React component state.
  - **Hiding UI safety**: **Hiding UI is NOT secure**. Hiding elements only improves user experience. Security must always be enforced at the backend API layer.

## 13. Edge Cases & Real-World Problems

### Must Ask / Critical Thinking

* **Employee departures & deleted managers**:
  - **Deleted Managers**: Use soft deletion flags (`isDeleted: true`) or reassign reporting profiles to a fallback admin.
  - **Duplicate/Overlapping Leaves**: The leave application schema contains a refinement constraint validating that `startDate` is not after `endDate`. The backend checks for overlapping date ranges in MongoDB before saving new requests:
    ```javascript
    const overlap = await Leave.findOne({
        user: userId,
        status: { $ne: 'rejected' },
        startDate: { $lte: newEndDate },
        endDate: { $gte: newStartDate }
    });
    if (overlap) throw new Error("Overlapping leave request already exists.");
    ```

## Panel Trick Questions

* **Authentication vs Authorization**:
  - **Authentication**: Confirming *who* you are (login credentials, tokens verification).
  - **Authorization**: Confirming *what* permissions you have (role checks, resource access restrictions).
* **Concurrent Approvals**: Resolving approvals concurrently can cause race conditions. Prevented by using MongoDB transaction boundaries or atomic `$set` operations.
* **Database Failures midway**: Wrapped in asynchronous error handling wrappers (`catchAsync`) to prevent thread hangs, clearing local tokens if credentials validation fails midway.
